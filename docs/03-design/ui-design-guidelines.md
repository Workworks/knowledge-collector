# UI 设计规范

## 1. 设计原则

清晰优先于装饰；状态可辨识；高频操作少跳转；阅读页减少管理界面噪音；错误信息必须可执行。

## 2. 色彩

| 用途 | 色值 |
| --- | --- |
| 主色 | `#2563EB` |
| 主色悬停 | `#1D4ED8` |
| 成功 | `#15803D` |
| 警告 | `#B45309` |
| 危险 | `#B91C1C` |
| 信息 | `#0369A1` |
| 正文 | `#1F2937` |
| 次级文字 | `#6B7280` |
| 边框 | `#D1D5DB` |
| 页面背景 | `#F8FAFC` |
| 卡片背景 | `#FFFFFF` |

所有状态同时使用文字或图标，不能只靠颜色。正文与背景对比度目标达到 WCAG AA。

## 3. 字体与排版

- 界面字体：系统无衬线字体栈，中文优先 `"Microsoft YaHei"`、`"PingFang SC"`。
- 基准字号：16px；辅助文字 14px；表格最低 14px。
- 标题：H1 28/36、H2 22/30、H3 18/26。
- 阅读正文：18px，行高 1.75，段间距 1em，最大宽度 760px。
- 代码与 URL：等宽字体并允许安全换行。

## 4. 间距、圆角与阴影

- 间距采用 4px 基准：4、8、12、16、24、32、48。
- 控件高度默认 40px，紧凑表格控件不低于 32px。
- 卡片与输入框圆角 8px，标签 999px。
- 卡片阴影轻量：`0 1px 3px rgba(15, 23, 42, .10)`。

## 5. 组件规范

### 5.1 按钮

主按钮每个操作区最多一个；次按钮用于并列操作；危险按钮使用红色并要求确认。按钮必须有可见焦点和禁用状态。

### 5.2 表单

标签置于输入框上方；必填项显示“*”及文本说明；帮助信息和错误信息位置稳定。URL、超时、重试、间隔提供格式与范围提示。

### 5.3 表格

表头固定语义、数值右对齐、状态居中、主要名称左对齐。窄屏优先转为卡片或允许横向滚动，不隐藏关键状态。

### 5.4 标签与状态徽章

主题标签使用管理员配置颜色但自动校验可读性。任务状态使用统一词汇：已创建、运行中、部分成功、成功、失败、已取消。

### 5.5 卡片与弹窗

卡片只聚合一种信息。弹窗用于确认或短表单，复杂编辑使用独立页面。关闭弹窗后焦点返回触发控件。

## 6. 图标

优先使用 Bootstrap Icons 或等价开源图标。图标按钮必须带 `aria-label`/可见提示；装饰图标对辅助技术隐藏。

## 7. 响应式规则

- `>=1200px`：固定侧栏 + 双栏内容。
- `768—1199px`：可折叠侧栏，表格保持。
- `<768px`：抽屉导航、单栏卡片、操作按钮可换行。
- 阅读页在所有尺寸保持正文舒适宽度，图片 `max-width:100%`。

## 8. 阅读页面规则

- 不嵌入第三方完整网页。
- 仅渲染已清洗 HTML。
- 标题、段落、列表、引用、代码和图片具有一致样式。
- 外部链接显示外链提示并在新标签页打开。
- 长 URL、代码和表格不能撑破页面。
- 深色阅读模式列为 P2，第一版不作为验收阻断。

## 9. 无障碍

- 所有交互可用键盘完成，焦点顺序符合视觉顺序。
- 表单错误通过文本和 `aria-describedby` 关联。
- 表格包含标题/表头；图表同时提供数据表或数值列表。
- 动画尊重 `prefers-reduced-motion`。
- 点击目标建议至少 40×40px。

## 10. 文案规范

使用用户语言描述动作，例如“测试来源”“立即采集”，避免暴露堆栈或内部类名。错误消息结构为“发生了什么 + 可能原因 + 下一步”，同时保留可用于日志关联的错误编号。
