# Knowledge Collector 项目立项说明书

## 1. 项目背景

个人与小团队每天面对大量分散在 RSS、Atom、公开网页、博客、研究机构和公开 API 中的信息。浏览器书签只能保存链接，难以持续采集、统一检索、去重、分类和形成阅读闭环。本项目拟建设一套本地单机部署的“爬虫资料收集与阅读管理系统”，将公开信息源的内容转化为可管理、可搜索、可阅读的个人资料库。

## 2. 项目愿景

在不依赖付费服务、不绕过网站访问控制的前提下，为管理员提供从“配置主题与来源”到“采集、清洗、分类、阅读、收藏和归档”的完整工作台。

## 3. 项目目标

1. 支持主题、采集源和采集规则的配置化管理。
2. 支持 RSS、Atom、HTML 列表、公开 JSON API 和手动 URL。
3. 建立可观测的手动与定时采集任务流程。
4. 实现 URL 与内容指纹去重、关键词主题匹配和可解释质量评分。
5. 提供文章搜索、筛选、阅读、收藏、已读、归档、标签和笔记功能。
6. 使用 Java 17、Spring Boot 3、H2、Flyway 和 Thymeleaf 实现本地一体化部署。
7. 建立自动化测试、文档、备份恢复和 Git 分阶段交付基线。

## 4. 目标用户与干系人

| 角色 | 关注点 | 责任 |
| --- | --- | --- |
| 系统管理员/主要用户 | 信息采集效率、内容质量、阅读体验、数据安全 | 配置来源、遵守来源条款、维护数据 |
| 开发维护者 | 可维护性、可测试性、可扩展性 | 设计、开发、测试、发布和修复 |
| 内容来源方 | 合规访问、署名、链接和版权边界 | 公布访问规则与内容许可 |

第一版只有 `ADMIN` 角色，可不实现登录，但代码需为后续鉴权预留边界。

## 5. 第一版本范围

第一版范围详见 [first-version-scope.md](first-version-scope.md)，核心包括：本地单用户管理、主题和来源配置、RSS/Atom/HTML/手动 URL 采集、任务管理、清洗去重、主题匹配、资料库、阅读状态、仪表盘、备份和本地部署。

## 6. 非目标

- 不建设微服务或分布式调度平台。
- 不绕过登录、验证码、付费墙、robots.txt 或反爬机制。
- 不提供代理池、身份伪造、批量账号或验证码破解能力。
- 不处理必须执行复杂 JavaScript 的动态页面；此类来源标记为 `UNSUPPORTED_DYNAMIC_PAGE`。
- 不将大语言模型、Elasticsearch、Redis、消息队列、Selenium 或 Playwright 作为第一版依赖。
- 不承诺替代专业版权管理、内容授权或企业级多租户系统。

## 7. 技术约束

- Java 17、Spring Boot 3.x、Maven、Spring MVC、Validation、Data JPA。
- H2 文件数据库，数据库结构统一由 Flyway 管理，JPA 使用 `ddl-auto=validate`。
- Thymeleaf、Bootstrap、原生 JavaScript，前后端一体化。
- Jsoup 负责 HTML 解析，Rome 负责 RSS/Atom。
- 本地文件系统保存运行数据；禁止在代码中写死绝对路径和采集站点。
- 外部请求必须设置超时、响应大小、重定向和有限重试限制。

## 8. 合规边界

系统只采集公开可访问且允许自动访问的内容，优先使用 RSS、Atom 和公开 API。默认遵守 robots.txt，同一来源请求间隔不少于两秒，保留来源名称与原文链接。管理员负责确认来源条款、版权许可和内容保留期限，并可停用来源、删除来源数据或仅保存摘要与链接。

## 9. 主要风险

主要风险包括来源结构变化、版权或 robots.txt 限制、页面编码异常、H2 大文本搜索性能、清洗误删、来源限流、数据文件损坏和需求范围膨胀。详细登记、应对措施和责任见 [risk-register.md](risk-register.md)。

## 10. 里程碑

| 阶段 | 里程碑 | 主要成果 |
| --- | --- | --- |
| Stage 1 | 项目与需求基线 | 立项、需求、范围、风险、假设、计划 |
| Stage 2 | 设计基线 | 架构、原型、UI、数据库、API、Maven 骨架 |
| Stage 3 | 可启动基线 | Spring Boot、H2、Flyway、日志、脚本 |
| Stage 4—7 | 采集核心闭环 | 配置、任务、RSS/Atom/HTML、清洗、分类 |
| Stage 8—9 | 阅读与运维闭环 | 资料库、阅读状态、调度、仪表盘 |
| Stage 10 | 发布验收 | 测试、打包、部署、备份恢复、手册 |

详细计划见 [iteration-plan.md](iteration-plan.md)。

## 11. 验收标准

Stage 1 验收以真实文件、Git 提交和标签为准；最终版本以需求规格中的可追踪需求和总体验收清单为准。任何未执行的编译、测试、页面验证或联网采集不得写为成功。

## 12. 项目治理

- 采用阶段门禁：上一阶段验收后才能进入下一阶段。
- 每阶段创建 `docs/stages/stage-{id}-report.md`。
- 数据库变更必须走 Flyway；关键规则必须有自动化测试。
- 需求变更需更新需求编号、追踪关系、风险和阶段报告。
- Git 提交信息使用“阶段/类型: 说明”的简洁格式，阶段完成创建对应标签。
