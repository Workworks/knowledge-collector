# 项目总体迭代计划

## 1. 迭代原则

- 严格按 Stage 1—10 顺序推进，阶段验收未通过不进入下一阶段。
- 每阶段先复核已有文档和上一阶段报告，再实现、测试、构建、更新文档、提交和打标签。
- 阶段结果只记录真实执行情况，不以计划代替完成事实。

## 2. 阶段计划

| 阶段 | 目标 | 主要交付物 | 退出条件 |
| --- | --- | --- | --- |
| 1 | 项目立项与需求基线 | 立项、用户需求、PRD、SRS、范围、风险、假设、计划 | Markdown 检查通过，Git 提交和 `stage-1` 标签 |
| 2 | 架构、原型与设计基线 | 原型、UI、架构、详细设计、数据库、API、Maven 骨架 | 设计评审通过，骨架结构可解析 |
| 3 | 基础工程与本地运行 | Boot、H2、Flyway、日志、异常、脚本 | 应用启动，数据库持久化，基础测试通过 |
| 4 | 主题与采集源管理 | CRUD、页面、REST API、测试 | 主题与来源闭环可验证 |
| 5 | RSS 与 Atom 采集 | Provider、任务、URL 标准化、去重、入库 | 固定样例采集测试通过 |
| 6 | HTML 网页采集 | 规则、预览、正文抽取与安全清洗 | HTML 样例与 XSS 测试通过 |
| 7 | 主题与质量控制 | 分类、排除、多主题、质量分、指纹 | 规则测试和待审核流程通过 |
| 8 | 资料库与阅读 | 搜索筛选、详情、收藏、已读、归档、笔记 | 用户阅读闭环通过 |
| 9 | 调度与运维 | 定时、重试、限速、健康、仪表盘、备份 | 调度与运维场景验证通过 |
| 10 | 测试、部署与验收 | 完整测试、手册、打包、备份恢复 | `mvn clean test/package` 与验收清单通过 |

## 3. 需求追踪

需求以 SRS 中的 `FR`、`NFR`、`SEC`、`DATA` 编号为主键。每个后续阶段报告必须列出本阶段覆盖的需求编号、测试证据和未关闭差距。

## 4. Git 计划

- 分支默认使用 `master`（仓库初始化现状）；如需改为 `main`，在进入 Stage 2 前统一调整并记录。
- 阶段提交格式：`stage-N: 阶段成果说明`。
- 阶段标签格式：`stage-N`。
- 功能开发阶段可增加 `feat`、`fix`、`test`、`docs` 类型提交，但阶段报告需汇总。

## 5. 质量门禁

1. 文档与实现一致。
2. 关键业务规则有自动化测试。
3. 外部网络不作为自动化测试成功的前提。
4. 安全和合规要求不可用“后续优化”规避。
5. 任何构建或测试失败必须记录原因和修复建议。
