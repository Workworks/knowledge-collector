# 用户需求文档

## 1. 用户画像

### 1.1 主要用户：知识资料管理员

个人研究者、开发者、产品人员或小团队资料管理员。具备基本网页和 RSS 概念，希望通过图形界面维护主题与来源，不希望搭建复杂基础设施。

### 1.2 维护者

负责安装、升级、备份、排错和扩展采集 Provider 的开发或技术人员。

## 2. 用户目标

- 用一个本地系统持续收集多个公开来源。
- 按主题获得经过清洗和去重的文章，而非重复链接。
- 快速找到未读、高质量或收藏内容。
- 看清采集是否成功、失败原因是什么。
- 保持数据在本机，能够备份、恢复和迁移。
- 以合规、克制的频率访问来源。

## 3. 核心使用场景

1. 管理员创建“人工智能”主题并设置关键词和排除词。
2. 管理员添加 RSS/Atom 或公开网页来源，配置请求参数和抓取规则。
3. 管理员测试来源或规则，确认候选链接和正文预览。
4. 管理员手动执行来源采集，或等待定时任务。
5. 系统记录发现、新增、重复、失败和跳过数量。
6. 管理员在资料库按主题、来源、状态和质量分筛选并搜索。
7. 用户阅读安全清洗后的正文，收藏、标记已读、归档或添加笔记。
8. 管理员查看失败任务、来源健康度并有限重试。
9. 维护者备份数据库和内容目录，并能按文档恢复。

## 4. 用户故事

| ID | 用户故事 | 验收要点 |
| --- | --- | --- |
| US-001 | 作为管理员，我希望创建和停用主题，以便控制资料分类 | 可保存必填字段；停用后不参与自动采集 |
| US-002 | 作为管理员，我希望配置不同类型来源，以便统一管理信息入口 | 支持五类来源并校验 URL、超时和重试 |
| US-003 | 作为管理员，我希望先测试规则，以避免错误配置污染资料库 | 展示提取预览，不直接入正式文章库 |
| US-004 | 作为管理员，我希望查看任务计数和错误，以便定位问题 | 有状态、耗时、计数、错误码和错误消息 |
| US-005 | 作为读者，我希望搜索和组合筛选文章，以便快速找到内容 | 分页、排序与筛选条件可组合 |
| US-006 | 作为读者，我希望阅读干净安全的正文，以避免广告和脚本干扰 | 不嵌入完整第三方页面，不执行危险 HTML |
| US-007 | 作为读者，我希望收藏、已读、归档和写笔记，以便维护阅读工作流 | 状态修改持久化且可撤销（归档可恢复） |
| US-008 | 作为管理员，我希望系统去重，以避免同一内容反复出现 | 标准 URL 与内容哈希均参与判断 |
| US-009 | 作为维护者，我希望离线启动和本地备份，以便不依赖云服务 | 启动不要求联网；备份步骤明确可验证 |

## 5. 业务规则

- 来源与规则必须配置化，禁止在代码中写死站点。
- 一个主题可关联多个来源，一篇文章可属于多个主题。
- 重复内容不得反复创建新文章；内容更新需增加版本并保留时间。
- 动态页面、登录页面、验证码和付费墙不尝试绕过。
- 默认遵守 robots.txt，并限制请求频率、响应大小和重试次数。
- 阅读页仅展示清洗后的安全 HTML，并保留来源与原文链接。

## 6. 数据与隐私期望

- 业务数据默认保存在用户本机。
- 不采集个人隐私数据，不记录敏感 Token 或完整 Cookie。
- 用户可备份、恢复、导出或清理本地数据。
- 删除来源时必须提供关联数据处理策略，避免误删。

## 7. 易用性与可访问性期望

- 默认中文界面，导航和状态名称一致。
- 表单提供必填提示、格式错误和操作结果反馈。
- 文章排版宽度、字号和行高适合长时间阅读。
- 颜色不作为唯一状态提示，关键控件可键盘操作。
- 空状态、加载状态和失败状态给出下一步操作。

## 8. 用户验收条件

最终用户能在本地完成“创建主题 → 配置来源/规则 → 执行采集 → 查看任务 → 搜索阅读 → 收藏/已读/归档/笔记 → 备份”的闭环，且系统不包含绕过访问限制的能力。
